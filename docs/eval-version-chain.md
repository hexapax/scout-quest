# Eval Version Chain: Version Provenance Tracking

This is the authoritative document on how versions flow through the Scout Quest evaluation system. Every link in the chain can change independently; the version fingerprint captures the state of every link at eval time.

## The Version Chain

```
Source Material (BSA publications, Scoutbook data)
  -> scout-corpus repo (extraction, enrichment pipelines)
    -> Knowledge Doc (interim-bsa-knowledge.md, committed to scout-quest)
      -> System Prompt (knowledge + persona + troop context, assembled at runtime)
        -> Model Response (generated by model under eval)
          -> Panel Evaluation (assessors observe, scorer judges)
            -> Eval Result (stored in MongoDB with version fingerprint)
```

Each arrow represents a transformation where information can be gained, lost, or distorted. Version tracking ensures that when a score changes, you can trace which link in the chain changed and whether the score delta is real or an artifact.

## What Gets Versioned

| Component | File(s) | How tracked | Change frequency |
|-----------|---------|-------------|------------------|
| BSA Knowledge | backend/knowledge/interim-bsa-knowledge.md | Content hash + git commit | Rare (when corpus updated) |
| Troop Context | backend/knowledge/troop-context.md | Content hash + git commit | Occasional (Scoutbook refresh) |
| Model Personas | backend/experiments/model-personas.json | Content hash + git commit | Rare (persona tuning) |
| Eval Set (questions) | eval-sets/*.yaml | Content hash + git commit | Moderate (adding questions, refining eval_notes) |
| Assessor Config | In eval set YAML | Content hash | When assessor models/prompts change |
| Scorer Prompt | In eval set YAML | Content hash | When scoring methodology changes |
| Run Config | eval-sets/configs.yaml | Content hash + git commit | When model/layer configs change |
| Chain Harness | mcp-servers/scout-quest/test/harness.ts | Content hash + git commit | When chain execution logic changes |
| Eval Runner | scripts/run-eval.py | Git commit (HEAD) | When runner logic changes |

### Component Details

**BSA Knowledge** (`backend/knowledge/interim-bsa-knowledge.md`): The core knowledge document generated from the `scout-corpus` repo. Contains BSA handbook content, merit badge requirements, advancement procedures. When regenerated, the file header should include the source commit: `_Generated from scout-corpus@a1b2c3d on 2026-03-22._`

**Troop Context** (`backend/knowledge/troop-context.md`): Troop-specific information from Scoutbook — roster, advancement status, upcoming events. Updated when Scoutbook data is refreshed.

**Model Personas** (`backend/experiments/model-personas.json`): Character definitions (Woody archetype) that shape the coaching style. Changes here affect coaching and troop_voice scores.

**Eval Set** (`eval-sets/*.yaml`): Questions, eval_notes, assessor prompts, scorer prompts, and scoring dimensions. The eval set is the evaluation instrument — changes here change what "good" means.

**Run Config** (`eval-sets/configs.yaml`): Model selection, layer configuration (knowledge/persona/troop-context inclusion), and model parameters (temperature, thinking mode). Defines the system under evaluation.

**Chain Harness** (`mcp-servers/scout-quest/test/harness.ts`): The TypeScript harness that drives multi-turn chain evaluations. Changes here affect how the model is invoked, which tools are available, and how tool calls are processed.

**Eval Runner** (`scripts/run-eval.py`): The orchestrator that loads configs, runs perspectives, and stores results. Changes here can affect how inputs are assembled and how results are recorded.

## The Version Fingerprint

Every eval run stores a `versions` dict in both MongoDB and the `meta.json` output file. This fingerprint captures the complete state of the system at eval time.

### Structure

```json
{
  "versions": {
    "git_head": "a1b2c3d",
    "git_dirty": "M backend/knowledge/interim-bsa-knowledge.md",

    "knowledge": {
      "content_hash": "f8e2a1b3",
      "git_commit": "c3d4e5f6"
    },
    "troop_context": {
      "content_hash": "1a2b3c4d",
      "git_commit": "d4e5f6a7"
    },
    "personas": {
      "content_hash": "9f8e7d6c",
      "git_commit": "e5f6a7b8"
    },
    "eval_set": {
      "content_hash": "5b4a3c2d",
      "git_commit": "f6a7b8c9"
    },
    "configs": {
      "content_hash": "2c3d4e5f",
      "git_commit": "a7b8c9d0"
    },
    "harness": {
      "content_hash": "7d6e5f4a",
      "git_commit": "b8c9d0e1"
    },

    "system_fingerprint": "a3f8c2e1",
    "eval_fingerprint": "d7b4e9f2"
  }
}
```

### Field Definitions

**`git_head`**: The commit hash of HEAD in the scout-quest repo at the time the eval was run. This is the broadest version anchor — it tells you the exact repo state.

**`git_dirty`**: Output of `git status --porcelain` for versioned files. If non-empty, the working tree had uncommitted changes and the results may not be reproducible from the commit alone. The eval runner warns when this is non-empty.

**`content_hash`**: `sha256(file_content)[:8]` — the first 8 hex characters of the SHA-256 hash of the file's contents. This captures the exact file state regardless of git history. Two runs with the same content hash for a component used the same content, even if the git commit differs (e.g., after a no-op reformat).

**`git_commit`**: The last git commit that touched a specific file (from `git log -1 --format=%h -- <file>`). This tells you when the file last changed, which is useful for identifying which commit introduced a change.

**`system_fingerprint`**: A composite hash of all system-under-eval components: knowledge + troop_context + personas + configs (model/layer selection). Two runs with the same system_fingerprint evaluated the same system configuration.

**`eval_fingerprint`**: A composite hash of all evaluation system components: eval_set + assessor config + scorer prompt + harness. Two runs with the same eval_fingerprint used the same evaluation instrument.

## Comparability Rules

These rules determine when eval scores can be meaningfully compared:

| Condition | Interpretation | Action |
|-----------|---------------|--------|
| Same `eval_fingerprint` | Same evaluator was used | Scores are directly comparable |
| Different `eval_fingerprint` | Evaluator changed | Scores may not be comparable — check what changed |
| Same `system_fingerprint` | Same system under eval | Score differences reflect evaluator variance, not system changes |
| Different `system_fingerprint` | System changed | Score differences may reflect real system changes |
| `git_dirty` is empty | Clean working tree | Results are reproducible from the commit |
| `git_dirty` is non-empty | Uncommitted changes present | Results may not be reproducible — warn user |

### Decision Matrix

| system_fingerprint | eval_fingerprint | What you can conclude |
|-------------------|-----------------|----------------------|
| Same | Same | Repeated measurement — score difference is noise/variance |
| Different | Same | System changed — score difference may be real improvement or regression |
| Same | Different | Evaluator changed — score difference is evaluator drift, not system change |
| Different | Different | Both changed — cannot attribute score difference to either; avoid this |

### Practical Guidance

When comparing two eval runs:

1. Check `eval_fingerprint` first. If different, stop — the scores are not directly comparable. Look at `eval-changelog.md` to understand what changed in the evaluator.
2. If `eval_fingerprint` matches, check `system_fingerprint`. If different, the score delta likely reflects a real system change. Drill into individual component hashes to identify what changed.
3. If both fingerprints match, you're measuring variance. Multiple such runs give you confidence intervals.
4. Always check `git_dirty`. If non-empty on either run, the results carry an asterisk — the exact system state may not be reconstructible.

## Why This Matters

### Compliance and Auditing

For a youth-facing AI product, you need to demonstrate that safety evaluations are traceable, reproducible, and tied to specific system versions. If a safety score drops, you need to prove what changed and when. The version fingerprint provides a complete audit trail: given any eval result, you can reconstruct the exact system configuration that produced it (provided `git_dirty` was empty).

### Development Steering

The eval system's value depends on knowing that score changes reflect real system changes, not evaluator drift. Without version tracking, you can't tell if a coaching score improved because the persona got better or because the scorer prompt got easier. The separate `system_fingerprint` and `eval_fingerprint` make this distinction explicit.

### Cross-Session Continuity

Different developers (or AI sessions) run evals at different times. Version fingerprints ensure everyone can reason about which results are comparable without needing to remember what changed between sessions. The fingerprint is self-describing — you don't need session context to interpret it.
